# Cube × Math 架构改造设计方案

## 零、当前架构诊断

### 0.1 文件结构现状

```
src/
├── App.tsx                    # 路由定义（6条路由）
├── main.tsx                   # 入口
├── components/
│   ├── Layout.tsx             # 全局布局（Navbar + Outlet + Footer）
│   ├── Navbar.tsx             # 导航栏（4个主链接）
│   ├── Footer.tsx             # 版权栏
│   ├── AnalogyNote.tsx        # 类比卡片（未被导入使用）
│   ├── MatrixDisplay.tsx      # 3×3矩阵可视化
│   └── cube3d/
│       └── Cube3DView.tsx     # Three.js 3D魔方组件（540行）
├── pages/
│   ├── Landing.tsx            # 首页
│   ├── Chapter.tsx            # 6章内容页（1084行，巨型文件）
│   ├── Tutorial.tsx           # 交互引导页（650行）
│   ├── Playground.tsx         # 实操训练厅
│   └── Archive.tsx            # 数学档案馆
├── lib/
│   ├── cubeLogic.ts           # 魔方数据逻辑
│   └── matrixMath.ts          # 矩阵运算库
├── store/
│   └── cubeStore.ts           # Zustand全局状态
└── types/
    ├── cube.ts                # 魔方类型定义
    └── math.ts                # 矩阵类型定义
```

### 0.2 核心问题

| 问题 | 严重度 | 根因 |
|------|:------:|------|
| Chapter.tsx 1084行 | 高 | 6章的所有 demo 组件（InvariantDemo、CommutatorDemo、ComposeVerify、InverseDemo、DirectionIndexCard、QuizOptions）+ 6章的文本内容数据全在一个文件里 |
| Demo组件不可复用 | 高 | 所有交互demo都是Chapter.tsx内部的`function`，Tutorial和Playground要用也调不到 |
| Section 5 InvariantDemo功能单薄 | 高 | 只有"随机打乱"+观察，缺失奇偶性和色向两个核心不变量的交互演示 |
| Section 6只有交换子demo | 高 | 缺少逆序撤销和共轭变换的交互演示——而这两个才是第6节keyPoints写的核心工具 |
| 第3章缺矩阵验证 | 中 | ComposeVerify只验证了"结果不同"，没有验证"矩阵乘积描述了魔方操作" |
| Playground缺提示系统 | 中 | 无AI/算法辅助，零基础用户面对打乱魔方完全无从下手 |
| Chapter.tsx 文本与组件耦合 | 中 | 修改措辞需要改 1084 行的Chapter.tsx，容易引入bug |
| 记法条重复5次 | 低 | `R=右面 U=上面 F=前面 '=逆时针` 这段JSX在5个章节条件分支中重写了5次 |

### 0.3 数据流现状

```
Chapter.tsx (1084行)
├── CHAPTER_CONTENT[6章] ──→ 左侧概念讲解区
├── CHAPTER_SUMMARY[6章] ──→ 底部小结
├── InvariantDemo ──→ 右侧交互区（第5章）
├── CommutatorDemo ──→ 右侧交互区（第6章）
├── ComposeVerify  ──→ 右侧交互区（第3章）
├── InverseDemo    ──→ 右侧交互区（第4章）
└── Cube3DView ──→ cubeStore (Zustand)

Tutorial.tsx (650行)
├── Cube3DView ──→ cubeStore
└── cubeStore.cubies ──→ isSolved() 检测

Playground.tsx
└── Cube3DView ──→ cubeStore.cubies/history/snapshots
```

**问题**：三个页面各自独立与 Cube3DView + cubeStore 交互，没有共享的"操作-验证-提示"抽象层。如果要给 Playground 加 AI 提示，需要从零写状态分析逻辑。

## 二、组件接口设计

### 2.1 Demo 组件统一接口

所有 demo 组件遵循相同模式：

```typescript
interface DemoProps {
  cubeRef: React.RefObject<Cube3DViewHandle | null>
  // ComposeVerify 使用两个 ref：
  // leftCubeRef + rightCubeRef 替代 cubeRef
}

// 每个 Demo 组件内部管理：
// - 自身状态（playing, step, count 等）
// - 动画生命周期（setTimeout链，cleanup）
// - 不直接修改 cubeStore（通过 Cube3DView.executeMove 间接操作）
```

### 2.2 InvariantDemo 升级为三模式

```typescript
type InvariantMode = 'centers' | 'parity' | 'orientation'

interface InvariantDemoProps {
  cubeRef: React.RefObject<Cube3DViewHandle | null>
}

// 模式1 centers：中心块不变（现有功能）
// 模式2 parity：显示角块/棱块排列奇偶性，每次打乱后自动判断是否相等
// 模式3 orientation：显示角块色向总和mod3、棱块色向总和mod2，验证始终为零
```

```typescript
// lib/parity.ts 新增函数
export function getCornerParity(cubies: Cubie[]): number    // 0=偶, 1=奇
export function getEdgeParity(cubies: Cubie[]): number      // 0=偶, 1=奇
export function getCornerOrientation(cubies: Cubie[]): number // mod 3
export function getEdgeOrientation(cubies: Cubie[]): number   // mod 2
```

### 2.3 hintEngine 接口

```typescript
// lib/hintEngine.ts
export type HintLevel = 'beginner' | 'intermediate'

export interface Hint {
  step: number           // 当前阶段编号
  stepName: string       // "白色十字" | "白色角块" | "中层棱块" ...
  notation: string       // 建议的下一步操作，如 "R'"
  explanation: string    // 为什么这样做
}

export function getHint(cubies: Cubie[], history: Move[], level: HintLevel): Hint
```

## 五、风险控制

| 风险 | 缓解措施 |
|------|----------|
| Phase 0 拆分后功能回归 | 每完成一步运行 `npx vite build`，确保零错误 |
| InvariantDemo 模式切换导致 store 污染 | 使用 `disableStore` prop + `cubeRef.current?.reset()` 清理 |
| hintEngine 对零基础用户的提示不够好 | 先实现初学者级别（只判断当前阶段+给出建议公式），进阶级别延后 |
| Tutorial 步骤扩展影响现有功能 | 只新增黄色十字 Phase，不修改已有角块/棱块逻辑 |
| 文件太多导致构建变慢 | 所有新组件 <200行，vite HMR 不受影响 |
